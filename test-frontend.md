# Тестовое задание, Frontend стажёр

Ориентировочное время: около 8 часов. Оценка «если делать спокойно, не забивая на мелочи». Если больше, потому что что-то учил по ходу, напиши в сопроводительном письме реальные часы.

Задание самодостаточное. Свой бэкенд писать не надо, используем публичный API dummyjson.com.

## Контекст

Мы делаем аналитическую платформу для ВШЭ. Фронт у нас на React 18 с Vite, plain CSS без UI-фреймворков, авторизация через JWT с автоматическим рефрешем, тёмная тема, таблицы с поиском и пагинацией, графики, xlsx-экспорт. Тестовое покрывает те же скиллы, но на более простом домене.

## Стек

Обязательно:

```
React 18 или 19 (как удобнее)
Vite
react-router-dom v7
plain CSS, без UI-фреймворков
fetch или axios
```

Запрещено: Next.js, Gatsby, Remix, CRA. Запрещено Tailwind, MUI, Chakra, Bootstrap, styled-components, Emotion. Никаких UI-китов.

Почему такое ограничение: у нас в проде plain CSS, и нам нужно увидеть, как ты пишешь стили руками, а не подбираешь классы из библиотеки. Если придёшь в команду, будешь писать так же.

Можно: Vitest для юнит-тестов, Playwright для e2e, dayjs или date-fns для дат, clsx для условных классов. Любые маленькие утилитарные библиотеки ок.

## Задача

Интерфейс каталога товаров с авторизацией, поиском и корзиной. Данные из `https://dummyjson.com`, это публичный бесплатный API без регистрации.

### Про CORS dummyjson (важно, прочитай до старта)

Dummyjson часто теряет заголовок `Access-Control-Allow-Origin` на не-2xx ответах (401, 429, 500). Preflight проходит, GET на 200 проходит, а как только сервер решает вернуть ошибку, CORS-заголовков в ответе нет, и браузер ругается. Это особенность публичной песочницы, не баг твоего кода.

Лечится vite dev-proxy. В `vite.config.js`:

```js
export default defineConfig({
  plugins: [react()],
  server: {
    proxy: {
      '/dummyjson': {
        target: 'https://dummyjson.com',
        changeOrigin: true,
        rewrite: (p) => p.replace(/^\/dummyjson/, ''),
      },
    },
  },
})
```

И в коде используешь `/dummyjson/auth/login` вместо `https://dummyjson.com/auth/login`. В дев-режиме Vite сам проксирует, CORS в принципе не появляется. В production-сборке надо либо держать такую же проксю на своём бэкенде, либо согласиться, что редкие 4xx будут выглядеть как CORS. Для тестового достаточно dev-прокси, в README опиши ситуацию.

### API, с которым работаешь

**Авторизация:**

```
POST https://dummyjson.com/auth/login
body: {"username": "emilys", "password": "emilyspass", "expiresInMins": 30}
response 200: {
  "id": 1,
  "username": "emilys",
  "email": "emily.johnson@x.dummyjson.com",
  "firstName": "Emily",
  "lastName": "Johnson",
  "image": "https://dummyjson.com/icon/emilys/128",
  "accessToken": "eyJhbGc...",
  "refreshToken": "eyJhbGc..."
}
response 400: {"message": "Invalid credentials"}
```

```
POST https://dummyjson.com/auth/refresh
body: {"refreshToken": "...", "expiresInMins": 30}
response: {"accessToken": "...", "refreshToken": "..."}
```

```
GET https://dummyjson.com/auth/me
header: Authorization: Bearer <accessToken>
response: тот же объект пользователя
```

Демо-аккаунты можно показать в UI: `emilys / emilyspass`, `michaelw / michaelwpass`, `sophiab / sophiabpass`. Полный список в `https://dummyjson.com/users`.

**Товары:**

```
GET https://dummyjson.com/products?limit=20&skip=0
GET https://dummyjson.com/products/search?q=phone&limit=20&skip=0
GET https://dummyjson.com/products/categories
GET https://dummyjson.com/products/category/smartphones
GET https://dummyjson.com/products/{id}
```

В объекте товара есть массив `reviews`: каждый отзыв это `rating` (целое от 1 до 5), `comment`, `reviewerName`, `reviewerEmail`, `date` в ISO. Пример отзыва: `{"rating": 2, "comment": "Would not recommend!", "date": "2024-05-23T08:56:21.618Z", "reviewerName": "John Doe"}`. Тексты английские, это нормально.

**Корзина:**

```
POST https://dummyjson.com/carts/add
body: {"userId": 1, "products": [{"id": 1, "quantity": 2}]}
response: созданная корзина с total, discountedTotal, items

GET https://dummyjson.com/carts/user/{userId}
response: {"carts": [...]}
```

Полная документация: `https://dummyjson.com/docs`.

### Страницы и роутинг

```
/login              форма входа
/                   каталог товаров (требует auth)
/products/:id       карточка товара (требует auth)
/cart               корзина (требует auth)
/profile            профиль плюс история корзин (требует auth)
*                   404
```

Защищённые роуты оборачиваешь в компонент `<ProtectedRoute>`. Если пользователь не авторизован, редиректишь на `/login?next=<оригинальный_url>`. После успешного логина возвращаешь на `next`, если он был.

### Страница логина

Два поля: username и password. Кнопка «Войти». Ссылка «забыл пароль» ведёт на заглушку с текстом «обратитесь к администратору», сама фича не реализуется.

Внизу блок с демо-аккаунтами:

```
Демо-аккаунты:
  emilys / emilyspass (обычный юзер)
  michaelw / michaelwpass
  sophiab / sophiabpass
```

По клику на любой из них поля заполняются автоматически и форма сразу сабмитится. Удобно для проверяющих.

Валидация на клиенте: оба поля обязательные, пока пусто, кнопка disabled. Длина пароля не меньше 6.

Сабмит: кнопка во время запроса disabled, текст меняется на «Вхожу...». На 400 показываем «Неверные имя пользователя или пароль». На сетевую ошибку: «Не удалось подключиться». На Enter форма сабмитится.

После успеха токены в localStorage, юзер в контекст, редирект на `next` или `/`.

### Каталог товаров

Таблица или сетка карточек, на выбор. У нас в проде таблицы, но для товаров карточки тоже уместны. Главное чтобы на мобилке 375px не разваливалось.

На каждой карточке: картинка (thumbnail), название, цена, рейтинг звёздами, категория.

Над списком: поле поиска с debounce 400мс (дёргает `/products/search?q=`) и дропдаун категорий (один раз загружаешь через `/products/categories`).

Под списком: пагинация, показываешь «1-20 из 194», кнопки «назад» и «вперёд». Можно добавить выбор размера страницы, это бонус.

Пока грузится, показываешь скелетон (серые плейсхолдеры карточек) или спиннер. Пустой результат поиска подписывай «Ничего не найдено».

Клик по карточке ведёт на `/products/:id`.

### Карточка товара

Сверху: галерея изображений (первое крупно, остальные превью). Название, цена с учётом скидки (перечёркнутая оригинальная плюс актуальная), рейтинг, категория, бренд, остаток на складе, описание.

Кнопка «В корзину» со счётчиком количества. При нажатии POST на `/carts/add`, на успех показываем тост «Добавлено в корзину».

Ниже отзывы (массив `reviews` из объекта товара). Перед списком визуализация распределения оценок: пять столбцов от 1 до 5 звёзд, высота пропорциональна количеству отзывов с этой оценкой. SVG своими руками, без chart.js и подобного. Нам интересно посмотреть, как ты работаешь с примитивами.

Список отзывов: карточка с именем автора, датой, звёздами и текстом.

### Корзина

Список товаров, которые пользователь добавил через API. Каждая строка: превью, название, цена за штуку, количество, промежуточный итог. Внизу итоговая сумма и кнопка «Оформить заказ». По клику показываем тост «Заказ оформлен» и чистим локальное состояние корзины, это заглушка.

Если корзина пуста, показываем «Ваша корзина пока пуста» со ссылкой на каталог.

Удалять товары из корзины через dummyjson нельзя (API не поддерживает редактирование). Если сделаешь локальный фильтр, будет плюс.

### Профиль

Сверху: аватарка (`image` из user), имя, email.

Ниже «История корзин» (GET `/carts/user/{userId}`) карточками: дата создания, количество товаров, итоговая сумма.

Кнопка «Выйти» чистит localStorage и редиректит на `/login`.

### Авторизация на клиенте

Контекст `AuthContext`:

```ts
{
  user: {id, username, email, firstName, lastName, image} | null,
  isAuthenticated: boolean,
  isLoading: boolean,
  login: (username, password) => Promise<void>,
  logout: () => Promise<void>,
}
```

Токены `accessToken` и `refreshToken` храним в localStorage.

На первой загрузке, если `accessToken` есть, дёргай `/auth/me` для проверки. Если 401, попробуй `/auth/refresh`. Если и там 401, чисти токены и веди на `/login`.

HTTP-wrapper (или axios interceptor) автоматически добавляет `Authorization: Bearer <accessToken>` к каждому запросу на dummyjson. При 401 пытается рефрешнуть токен и ретрайнуть оригинальный запрос. Если после рефреша снова 401, делает logout, иначе получишь бесконечный цикл 401 → refresh → 401.

Учти гонку при параллельных запросах. Если десять компонентов одновременно получат 401, они не должны все десять стартовать рефреш. Простое решение это промис-мьютекс: первый запрос стартует рефреш, остальные ждут тот же промис. Для тестового можно пропустить (dummyjson всё равно не даёт токенам нормально истекать по времени), но плюс если реализуешь и напишешь в README.

### Тёмная тема

Переключатель в шапке, иконка солнца и луны. Выбор сохраняешь в localStorage.

На старте, если в localStorage пусто, смотришь `window.matchMedia('(prefers-color-scheme: dark)').matches` и используешь его. Это уважает системные настройки, пользователю не надо тыкать переключатель вручную.

Реализация через CSS-переменные:

```css
:root {
  --bg: #ffffff;
  --text: #1a1a1a;
  --accent: #0071e3;
  --border: #e5e5e5;
  --card-bg: #f9f9f9;
}

[data-theme="dark"] {
  --bg: #1a1a1a;
  --text: #f5f5f5;
  --accent: #3b9dff;
  --border: #333;
  --card-bg: #252525;
}

body {
  background: var(--bg);
  color: var(--text);
  transition: background 0.2s, color 0.2s;
}
```

Переключатель ставит атрибут `data-theme` на `<html>` или `<body>`. Никаких условных классов по компонентам, всё через переменные.

Чтобы тема не мигала при первой загрузке, в `index.html` до рендера React можно инлайн-скриптом прочитать localStorage и сразу поставить `data-theme` на `<html>`.

### Адаптивность

Проверяй на 375px (iPhone SE) и 1280px (ноут). Сетка карточек: на десктопе 4-5 колонок, на планшете 2-3, на мобилке 1. Модалки на мобилке занимают весь экран.

Минимум один breakpoint понадобится, обычно хватает 768px. Без media-queries не обойтись.

### Обработка ошибок

Вокруг приложения `ErrorBoundary` с fallback «Что-то пошло не так, обновите страницу» и кнопкой рефреша.

Сетевые ошибки и 5xx не должны ронять UI. Показывай сообщение «Не удалось загрузить товары» с кнопкой «Повторить».

401 триггерит refresh, не сразу logout. Только на повторный 401 после refresh делаешь logout.

## Бонусы

Всё необязательное, чем больше, тем лучше.

**Vitest-тесты.** Минимум один на `ProtectedRoute` (редирект без токена). Пара тестов компонентов через `@testing-library/react`: кнопка «Войти» disabled пока поля пустые, клик по карточке ведёт на детальную.

**Playwright e2e.** Один сценарий: логин, поиск товара, открытие карточки, добавление в корзину, переход в корзину, логаут.

**Виртуализация** сетки через `react-window`. Для 20 элементов это overkill, но покажет, что думаешь о перформансе.

**XLSX-экспорт** истории корзин через `xlsx` или `exceljs`. У нас в проде есть.

**Доступность.** Таб по элементам, Enter на кнопках, Escape закрывает модалки, видимый focus-ring, aria-label на иконочных кнопках, `role="status"` на лоадерах, семантические теги (button, a, nav, main вместо div).

**Lighthouse.** Chrome DevTools → Lighthouse. Цель: Performance, Accessibility, Best Practices выше 90. Скриншот в README.

**Умный лоадер.** Спиннер показывается только если запрос длится дольше 200мс, иначе на быстрых ответах он «мигает» и раздражает. Реализуется через `setTimeout` + `clearTimeout` в useEffect.

**Progressive enhancement.** Если JS отключён, хотя бы видно название сайта и подсказку «включите JS». Мало кто делает, но приятно.

## Частые затыки

**CORS errors в консоли.** См. раздел выше про dummyjson и vite proxy. Без прокси ты будешь биться с этим ещё час.

**Refresh бесконечно гоняется.** Interceptor пытается рефрешнуть токен, получает 401, снова пытается. Добавь флаг «уже пытались для этого запроса» или сделай правильный промис-мьютекс.

**Тема мигает при загрузке страницы.** React рендерится с светлой темой, потом useEffect читает localStorage и переключает. Решение: инлайн-скрипт в `<head>` до рендера, ставит `data-theme` на html до того, как увидел пользователь.

**Поиск дёргает API на каждой букве.** Забыл про debounce 400мс. Используй кастомный `useDebounce` или лодашовый.

## Критерии приёмки

Мы запускаем ровно так:

```
git clone <твой репо>
cd <папка>
npm install
npm run dev
```

Открываем `http://localhost:5173`, логинимся демо-аккаунтом, проходим флоу.

**Архитектура.** Компоненты разделены разумно, в `shared/` переиспользуемые кирпичики, нет копипасты одной и той же кнопки пять раз.

**Состояние.** Контекст не раздут до перерендеринга всего дерева на каждое действие. `useState`, `useMemo`, `useCallback` используются по делу, а не везде «на всякий случай».

**CSS.** Тема через переменные, нет инлайн-стилей с конкретными пикселями. Нет копипасты цветов. Классы именуются осмысленно (BEM или просто `component__element`, не `red-button-2`).

**UX-детали.** Форма не теряет введённое при ошибке сервера. Кнопка во время запроса disabled с лоадером. Успех показывается тостом, а не просто исчезновением формы.

**Обработка ошибок.** 5xx не крашит UI, показывает осмысленное сообщение с кнопкой «Повторить». 401 триггерит refresh. Пустой результат поиска показывает текст, а не пустое место.

**Доступность.** Навигация с клавиатуры работает. На картинках alt. Заголовки h1-h6 по смыслу, а не по размеру шрифта.

**Код.** Прошёл `npm run lint`. Если где-то отключал правило ESLint, оставил комментарий почему.
