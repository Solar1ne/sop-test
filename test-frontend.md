# Тестовое задание — Frontend, стажёр

Ориентировочное время: около 8 часов. Оценка «если делать спокойно и не забивать». Дольше тоже ок, если честно пишешь сколько ушло.

Задание самодостаточное: никакой свой бэкенд писать не нужно, используем публичный API dummyjson.com.

## Контекст

Мы делаем аналитическую платформу для ВШЭ. Фронт у нас на React 19 с Vite, plain CSS без UI-фреймворков, авторизация через JWT с автоматическим рефрешем, тёмная тема, таблицы с поиском и пагинацией, графики, xlsx-экспорт. Тестовое покрывает все эти скиллы, но на более простом домене.

## Стек

Обязательно:

```
React 19
Vite
react-router-dom v7
plain CSS (без фреймворков)
fetch или axios для HTTP
```

Запрещено: Next.js, Gatsby, Remix, CRA. Запрещено Tailwind, MUI, Chakra, Bootstrap, styled-components, Emotion. Никаких UI-китов.

Можно: Vitest для тестов, Playwright для e2e, dayjs или date-fns для дат, clsx для условных классов. Любые маленькие утилитарные библиотеки.

## Задача

Сделай интерфейс для каталога товаров с авторизацией, поиском и корзиной. Данные тянешь из `https://dummyjson.com` (публичный бесплатный API, не требует регистрации).

### API, с которым работаешь

**Авторизация**:

```
POST https://dummyjson.com/auth/login
body: {"username": "emilys", "password": "emilyspass", "expiresInMins": 30}
response 200: {
  "id": 1,
  "username": "emilys",
  "email": "emily.johnson@x.dummyjson.com",
  "firstName": "Emily",
  "lastName": "Johnson",
  "gender": "female",
  "image": "https://dummyjson.com/icon/emilys/128",
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
response 400: {"message": "Invalid credentials"}
```

```
POST https://dummyjson.com/auth/refresh
body: {"refreshToken": "...", "expiresInMins": 30}
response: {"accessToken": "...", "refreshToken": "..."}
```

```
GET https://dummyjson.com/auth/me
header: Authorization: Bearer <accessToken>
response: тот же объект пользователя
```

Тестовые логины (можно показать в UI как demo-аккаунты): `emilys / emilyspass`, `michaelw / michaelwpass`, `sophiab / sophiabpass`. Список всех юзеров и паролей: `https://dummyjson.com/users`.

**Товары**:

```
GET https://dummyjson.com/products?limit=20&skip=0
GET https://dummyjson.com/products/search?q=phone&limit=20&skip=0
GET https://dummyjson.com/products/categories
GET https://dummyjson.com/products/category/smartphones
GET https://dummyjson.com/products/{id}
```

Структура продукта:
```json
{
  "id": 1,
  "title": "Essence Mascara Lash Princess",
  "description": "...",
  "category": "beauty",
  "price": 9.99,
  "discountPercentage": 7.17,
  "rating": 4.94,
  "stock": 5,
  "brand": "Essence",
  "images": ["https://...", "..."],
  "thumbnail": "https://...",
  "reviews": [
    {"rating": 2, "comment": "Ужасно", "reviewerName": "John Doe", "reviewerEmail": "...", "date": "2024-05-23"},
    {"rating": 5, "comment": "Отлично!", ...}
  ]
}
```

**Корзина** (каждый юзер может положить товары в корзину):

```
POST https://dummyjson.com/carts/add
body: {"userId": 1, "products": [{"id": 1, "quantity": 2}]}
response: созданная корзина с total, discountedTotal, items

GET https://dummyjson.com/carts/user/{userId}
response: {"carts": [...]}
```

Полная документация: `https://dummyjson.com/docs`. Есть эндпоинты для пользователей, постов, рецептов и прочего, можешь использовать если придумаешь как.

### Страницы и роутинг

```
/login              форма входа
/                   каталог товаров (требует auth)
/products/:id       карточка товара (требует auth)
/cart               корзина (требует auth)
/profile            профиль + история корзин (требует auth)
*                   404
```

Защищённые роуты через компонент `<ProtectedRoute>`. Если пользователь не авторизован, редирект на `/login?next=<оригинальный_url>`. После успешного логина возврат на `next`, если он был.

### Страница логина

Два поля: username и password. Под ними кнопка «Войти». Линк «забыл пароль» ведёт на заглушку с текстом «обратитесь к администратору» (фича не реализуется).

Внизу блок с demo-аккаунтами, например:

```
Демо-аккаунты:
  emilys / emilyspass (обычный юзер)
  michaelw / michaelwpass
```

По клику на любой из них поля заполняются автоматически и форма сразу сабмитится. Удобно для проверяющих.

Валидация на клиенте: оба поля обязательные, пустые сабмитятся кнопка disabled. Длина пароля не меньше 6 (dummyjson пароли все по 9 символов но пусть будет).

Сабмит: кнопка во время запроса disabled, текст «Вхожу...». На 400 показываем «Неверные имя пользователя или пароль». На сетевую ошибку: «Не удалось подключиться». На Enter форма сабмитится.

После успеха токены в localStorage, юзер в контекст, редирект на `next` или `/`.

### Каталог товаров

Таблица или сетка карточек, на выбор. У нас в проде таблицы, но для товаров карточки тоже уместны. Главное чтобы на мобилке (375px) не разваливалось.

На каждой карточке: картинка (thumbnail), название, цена, рейтинг со звёздами, категория.

Над списком: строка поиска (debounce 400мс, дёргает `/products/search?q=`), дропдаун категорий (загружаешь один раз через `/products/categories`).

Под списком: пагинация с показом «1-20 из 194». Кнопки «назад/вперёд», можно добавить выбор размера страницы.

Пока грузится, показываем скелетон (серые плейсхолдеры карточек) или спиннер. Пустой результат поиска подписываем «Ничего не найдено».

Клик по карточке ведёт на `/products/:id`.

### Карточка товара

Сверху: галерея изображений (первое крупно, остальные превью). Название, цена с учётом скидки (перечёркнутая оригинальная плюс актуальная), рейтинг, категория, бренд, остаток на складе, описание.

Кнопка «В корзину» со счётчиком количества. При нажатии POST `/carts/add`, на успех показываем тост «Добавлено в корзину».

Ниже блок с отзывами (`reviews` из объекта товара). Сразу после отзывов визуализация распределения оценок: пять столбцов от 1 до 5 звёзд, высота пропорциональна количеству отзывов с этой оценкой. SVG своими руками, без chart.js и подобного. Хотим посмотреть, как ты работаешь с примитивами.

Список отзывов: каждая строка это карточка с именем автора, датой, звёздами и текстом.

### Корзина

Список товаров, добавленных юзером через API. Каждая строка: превью, название, цена, количество, промежуточный итог. Внизу итоговая сумма и кнопка «Оформить заказ» (заглушка: по клику показываем тост «Заказ оформлен!» и чистим локальное состояние).

Если корзина пуста, показываем «Ваша корзина пока пуста» с линком на каталог.

Удаление товара из корзины можно не делать (dummyjson не даёт нормально редактировать корзину), но если сделаешь локальный фильтр, будет плюс.

### Профиль

Сверху: аватарка (`image` из user), имя, email.

Ниже «История корзин» (GET `/carts/user/{userId}`) с карточками: дата создания, количество товаров, общая сумма.

Кнопка «Выйти»: чистит localStorage, редирект на `/login`.

### Авторизация на клиенте

Контекст `AuthContext`:

```ts
{
  user: {id, username, email, firstName, lastName, image} | null,
  isAuthenticated: boolean,
  isLoading: boolean,
  login: (username, password) => Promise<void>,
  logout: () => Promise<void>,
}
```

Токены (`accessToken`, `refreshToken`) храни в localStorage.

На первой загрузке, если есть `accessToken`, дёрни `/auth/me` чтобы проверить его. Если 401, попробуй `/auth/refresh`. Если тоже 401, чисти токены и веди на `/login`.

HTTP-wrapper (или axios interceptor) автоматически добавляет `Authorization: Bearer <accessToken>` к каждому запросу на dummyjson.com. При 401 пробует рефреш, потом ретраит оригинальный запрос. Если рефреш не помог, логаут.

Учти гонку при параллельных запросах: если десять компонентов одновременно получили 401, они не должны все десять стартовать рефреш. Простое решение через promise-мьютекс: первый запрос стартует рефреш, остальные ждут его результата. Для тестового можно игнорировать эту проблему (dummyjson всё равно не даёт реально истечь токену), но плюс если подумал и написал.

### Тёмная тема

Переключатель в шапке (солнце/луна). Выбор в localStorage.

На старте, если в localStorage ничего нет, смотри `window.matchMedia('(prefers-color-scheme: dark)').matches` и используй его.

Через CSS-переменные:

```css
:root {
  --bg: #ffffff;
  --text: #1a1a1a;
  --accent: #0071e3;
  --border: #e5e5e5;
  --card-bg: #f9f9f9;
}

[data-theme="dark"] {
  --bg: #1a1a1a;
  --text: #f5f5f5;
  --accent: #3b9dff;
  --border: #333;
  --card-bg: #252525;
}

body {
  background: var(--bg);
  color: var(--text);
  transition: background 0.2s, color 0.2s;
}
```

Переключатель ставит `data-theme` на `<html>` или `<body>`. Никаких условных классов по компонентам, всё через переменные.

### Адаптивность

Проверяй на 375px (iPhone SE) и 1280px (ноут). Сетка карточек на десктопе 4-5 колонок, на планшете 2-3, на мобильном 1. Модалки на мобильном занимают весь экран.

Минимум один breakpoint нужен, обычно 768px хватает. Без media-queries не обойдёшься.

### Обработка ошибок

Вокруг приложения `ErrorBoundary` с fallback «Что-то пошло не так, обновите страницу» и кнопкой рефреша.

Сетевые ошибки и 5xx не должны ронять UI. Показывай сообщение «Не удалось загрузить товары» с кнопкой «Повторить».

На 401 триггерится refresh, не сразу logout. На повторный 401 после refresh уже logout.

## Бонусы

**Vitest-тесты**. Минимум один на `ProtectedRoute` (редирект без токена). Плюс пара тестов компонентов через `@testing-library/react`: проверка, что кнопка «Войти» disabled пока поля пустые, что клик по карточке ведёт на детальную.

**Playwright e2e**. Один сценарий: логин → поиск товара → открытие карточки → добавление в корзину → переход в корзину → логаут.

**Виртуализация** таблицы/сетки через `react-window`. Для 20 элементов overkill, но покажет, что думаешь о перфе.

**XLSX-экспорт** истории корзин через библиотеку `xlsx` или `exceljs`. У нас в проде есть.

**Доступность**. Tab по элементам, Enter на кнопках, Escape закрывает модалки, видимый focus-ring, aria-label на иконочных кнопках, `role="status"` на лоадерах, семантические теги.

**Lighthouse**. Chrome DevTools → Lighthouse. Цель: Performance, Accessibility, Best Practices выше 90. Скриншот в README.

**Progressive enhancement**. Если disabled JavaScript, хотя бы видно название сайта и call-to-action «включите JS». Мало кто делает, но приятно.

## Критерии приёмки

Мы запускаем:

```
git clone <твой репо>
cd <папка>
npm install
npm run dev
```

Открываем `http://localhost:5173`, логинимся под демо-аккаунтом, проверяем флоу.

**Архитектура**. Компоненты разделены разумно, есть переиспользуемые кирпичики в `shared/`, нет копипасты одной и той же кнопки пять раз.

**Состояние**. Контекст не раздут до перерендеринга всего дерева на каждое действие. `useState`, `useMemo`, `useCallback` используются по делу, а не везде «на всякий случай».

**CSS**. Тема через переменные, нет инлайн-стилей с пикселями. Нет копипасты цветов. Классы именуются осмысленно (BEM или просто `component__element`, не `red-button-2`).

**UX-детали**. Форма не теряет введённое при ошибке. Кнопка disabled и с лоадером во время запроса. Спиннер не мигает на быстрых запросах (скелетон появляется только если запрос дольше 200мс). Успех показывается тостом, а не просто исчезает форма.

**Обработка ошибок**. 5xx не крашит UI. 401 триггерит refresh. Пустой результат поиска показывает сообщение.

**Доступность**. Навигация с клавиатуры, alt на картинках, h1-h6 по смыслу.

**Код**. Прошёл `npm run lint`. Если отключал правило, оставь комментарий почему.
